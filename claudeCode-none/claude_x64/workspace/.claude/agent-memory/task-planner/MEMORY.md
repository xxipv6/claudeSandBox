## 任务拆解原则

### SMART 原则

每个子任务应该：
- **Specific**（具体的）：明确要做什么
- **Measurable**（可衡量的）：有清晰的完成标准
- **Achievable**（可实现的）：在能力范围内
- **Relevant**（相关的）：与整体目标相关
- **Time-bound**（有时限的）：可以估算时间

### 粒度控制

- **太粗**：一个子任务包含多个目标，难以执行
- **太细**：子任务太多，管理成本高
- **合适**：每个子任务可以在 30-120 分钟内完成

### 依赖识别

- **强依赖**：A 必须在 B 之前（B 需要 A 的输出）
- **弱依赖**：A 最好在 B 之前（但不是必须）
- **无依赖**：A 和 B 可以并行

## 常见任务类型

### 安全研究项目

**典型子任务**：
- T1: 固件提取与分析
- T2: 通信协议分析
- T3: Web 接口测试
- T4: 物理接口分析
- T5: 漏洞利用 PoC

**Agent 分配**：
- 固件/协议分析：backend-engineer + security-tester
- Web 测试：security-tester + frontend-engineer
- 物理接口：backend-engineer
- PoC 开发：script-coder

### 全栈开发项目

**典型子任务**：
- T1: 需求分析
- T2: 数据库设计
- T3: API 设计
- T4: 前端页面开发
- T5: 后端实现

**Agent 分配**：
- 需求分析：product-manager
- 数据库/API 设计：backend-engineer
- 前端开发：dev-coder
- 后端开发：dev-coder

### 漏洞分析项目

**典型子任务**：
- T1: 漏洞定位
- T2: 漏洞原理分析
- T3: 影响范围评估
- T4: PoC 编写
- T5: 修复建议

**Agent 分配**：
- 漏洞定位：security-tester + backend-engineer
- 原理分析：security-tester + backend-engineer
- 影响评估：security-tester + backend-engineer
- PoC 编写：script-coder
- 修复建议：backend-engineer + security-tester

## 依赖关系模式

### 串行依赖

```
T1 → T2 → T3
```

**示例**：
- T1: 需求分析
- T2: 数据库设计（依赖 T1）
- T3: API 设计（依赖 T1, T2）

### 并行执行

```
T1 ─┬→ T3
    │
T2 ─┘
```

**示例**：
- T1: 前端页面 A
- T2: 前端页面 B
- T3: 集成测试（依赖 T1, T2）

### 复杂依赖

```
T1 ─┬→ T3 ─┬→ T5
    │      │
T2 ─┘     T4 ─┘
```

**示例**：
- T1: 固件分析
- T2: 协议分析
- T3: 漏洞发现（依赖 T1, T2）
- T4: Web 测试
- T5: PoC 开发（依赖 T3, T4）

## 风险识别模式

### 常见风险

| 风险类型 | 影响 | 缓解措施 |
|---------|------|---------|
| 依赖失败 | 高 | 提前验证依赖，准备替代方案 |
| 资源不足 | 中 | 合理分配资源，分阶段执行 |
| 需求变更 | 高 | 明确边界，快速迭代 |
| 技术难题 | 中 | 提前调研，准备备选方案 |
| 时间估算错误 | 低 | 预留缓冲时间 |

### 风险评估

- **高影响 + 高概率**：必须优先处理
- **高影响 + 低概率**：制定应对计划
- **低影响 + 高概率**：接受风险
- **低影响 + 低概率**：可以忽略

### 分析层 Agent

|-------|---------|

## 输出规范

### 必须包含的内容

1. **目标理解**
   - 用户目标：明确的目标描述
   - 成功标准：可验证的完成条件
   - 范围边界：包含什么、不包含什么

2. **任务拆解**
   - 任务ID：T1, T2, T3...
   - 任务描述：清晰的任务描述
   - 输入：这个任务需要什么
   - 输出：这个任务产生什么
   - 复杂度：简单/中等/复杂
   - 预计时间：估算时间

3. **依赖关系**
   - 依赖图：可视化依赖关系
   - 关键路径：最长的执行路径
   - 并行组：可以同时执行的任务

4. **执行计划**
   - 阶段划分：串行阶段
   - 并行任务：每个阶段内的并行任务
   - 总时长：估算总时间

5. **Agent 分配**
   - 每个任务需要的 Agent
   - 为什么需要这个 Agent

6. **风险识别**
   - 潜在风险
   - 影响评估
   - 缓解措施

7. **待澄清问题**
   - 需要用户确认的问题
   - 可能影响规划的问题

## 常见陷阱

### ❌ 不要这样做

1. **任务太大**
   ```
   T1: 实现用户系统  # 太大，无法执行
   ```

2. **依赖关系错误**
   ```
   T3 → T1  # T3 依赖 T1，但 T1 在后面
   ```

3. **Agent 分配错误**
   ```
   T1: security-tester  # 安全分析任务分配给了前端工程师
   ```

4. **遗漏子任务**
   ```
   # 缺少测试阶段
   T1: 开发
   T2: 部署
   ```

### ✅ 应该这样做

1. **合适的任务粒度**
   ```
   T1: 设计用户表结构
   T2: 实现用户注册 API
   T3: 实现用户登录 API
   ```

2. **正确的依赖关系**
   ```
   T1: 需求分析
   T2: 数据库设计（依赖 T1）
   T3: API 实现（依赖 T2）
   ```

3. **正确的 Agent 分配**
   ```
   T1: backend-engineer, security-tester  # 安全分析任务
   ```

4. **完整的任务列表**
   ```
   T1: 开发
   T2: 测试
   T3: 部署
   T4: 验证
   ```
