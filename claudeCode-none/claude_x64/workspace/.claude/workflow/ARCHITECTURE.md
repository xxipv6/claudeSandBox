# 新系统架构说明

## 一、核心模式（单一系统，自动分级）

```
用户输入
    ↓
Claude 理解意图
    ↓
自动判断复杂度
    ↓
┌─────────┬─────────┬─────────┐
│  快速    │  标准    │  完整    │
│  模式    │  模式    │  模式    │
└─────────┴─────────┴─────────┘
```

---

## 二、三种执行模式对比

### 快速模式（Quick Mode）
```
输入：单文件操作 / 明确修复 / 用户说"快速"
  ↓
直接执行（0 个 agents）
  ↓
输出结果
```

**特点**：
- 无 agents
- 无状态文件
- 无计划确认
- 直接修改代码

**适用**：
- 单文件修改
- 明确的 bug 修复
- 用户说"快速"、"简单"

---

### 标准模式（Standard Mode）← 默认模式
```
输入：多文件操作 / 需要分析 / 用户说"分析"
  ↓
制定简要计划（2-3 个步骤）
  ↓
询问用户确认
  ↓
启动 1-2 个 agents（顺序执行）
  ↓
合并结果
  ↓
输出简洁报告
```

**特点**：
- 1-2 个 agents（顺序）
- 可选状态文件
- 简要计划确认
- 解决问题优先

**适用**：
- 项目分析
- 功能实现
- 代码审查
- **任何不确定的情况**（安全缓冲区）

---

### 完整模式（Full Mode）
```
输入：架构级任务 + 用户说"完整分析"
  ↓
检查环境变量 CLAUDE_FULL_MODE=1
  ↓
制定详细计划（6 个阶段）
  ↓
询问用户确认
  ↓
执行完整流程
  ↓
输出完整结果
```

**特点**：
- 2-4 个 agents（顺序）
- 完整状态文件
- 详细计划确认
- Git 分支管理
- Quality Gate

**适用**：
- 架构级安全研究
- 需要明确授权的场景

**触发条件**（必须同时满足）：
- 架构级任务
- 用户明确说"完整分析"
- 环境变量已设置

## 四、Agent 使用模式

### 之前的设计（复杂）
```
Analysis Mode:
  并发启动 4 个 agents
  ↓
  等待所有返回
  ↓
  合并结果
```

### 现在的设计（简化）
```
标准模式:
  选择 1-2 个相关 agents
  ↓
  顺序执行
  ↓
  合并结果
```

**为什么顺序执行？**
- Claude 的"并发"是叙事级的，不是系统级
- 顺序执行更稳定、更可控
- 更容易合并结果

## 五、判断逻辑（保守策略）

```python
def detect_mode(user_input, context):
    # 快速模式（明确信号）
    if is_single_file(context) or has_explicit_fix(user_input) or user_says_quick(user_input):
        return "quick"

    # 完整模式（明确授权）
    if is_architecture_level(context) and user_says_full_analysis(user_input) and env_has_full_mode():
        return "full"

    # 标准模式（默认，安全缓冲区）
    return "standard"  # 任何不确定都到这里
```

**核心原则**：不确定 → 标准模式

## 六、执行流程对比

### 之前的流程（660 行）
```
阶段 1：读取配置（8 个文件）
  ↓
阶段 1：启动 task-planner
  ↓
阶段 1：读取知识（4 个文件）
  ↓
阶段 2：执行 6 个 stages
  ↓
Stage 03：并发启动 4 个 agents
  ↓
输出 Research Ledger
```

### 现在的流程（109 行）
```
理解用户意图
  ↓
自动选择模式
  ↓
快速模式：直接执行
标准模式：简要计划 → 1-2 agents → 结果
完整模式：详细计划 → 2-4 agents → 完整流程
```

## 七、核心设计理念

| 维度 | 之前 | 现在 |
|-----|------|------|
| **驱动方式** | 配置驱动 | 意图驱动 |
| **复杂度** | 660 行规则 | 109 行原则 |
| **Agent 数量** | 固定 4 个并发 | 0-4 个顺序 |
| **模式选择** | 两阶段强制 | 自动判断 |
| **状态文件** | 强制创建 | 按需创建 |
| **用户确认** | 形式化 | 实用化 |
| **优先级** | 证明合规 | 解决问题 |
| **安全缓冲** | 无 | 标准模式 |

## 九、为什么这样设计？

### 问题 1：660 行无法执行
**解决**：精简到 109 行核心原则

### 问题 2：配置驱动不工作
**解决**：改为意图驱动（Claude 擅长理解意图）

### 问题 3：并发 agents 不可控
**解决**：改为顺序执行

### 问题 4：过度形式化
**解决**：优先解决问题，不是证明合规

### 问题 5：没有安全缓冲
**解决**：标准模式作为默认和安全缓冲区

## 十、系统性质

这是**人为定义的智能任务执行系统**：

- 不是 Claude Code 的官方支持模式
- 不是配置驱动系统
- 不是协议执行引擎
- **是**意图驱动的智能助手
- **是**根据任务复杂度自动调整的系统
- **是**优先解决问题的实用化设计
