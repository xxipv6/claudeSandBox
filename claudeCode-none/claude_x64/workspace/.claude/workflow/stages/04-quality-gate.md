# Stage 04: Quality Gate（质量门禁）

## 目标

质量验证，包括静态分析、安全扫描、自动测试。

## 触发条件

- Stage 03: Mode Execution 已完成
- 所有 agents 已执行完成

## 执行步骤

### 第一步：确定质量检查范围

根据任务类型确定质量检查项目：

```bash
# 代码任务
static_analysis=true
security_scan=true
tests=true

# 分析任务
static_analysis=false
security_scan=false
tests=false
```

**检查点**：
- [ ] 质量检查范围已确定
- [ ] 检查项目已列出

### 第二步：静态分析

对生成的代码进行静态分析：

```bash
# 检查代码语法
# 检查代码风格
# 检查代码结构
```

**检查点**：
- [ ] 静态分析已完成（如适用）
- [ ] 问题已记录
- [ ] 修复建议已列出

### 第三步：安全扫描

对生成的代码进行安全扫描：

```bash
# 检查安全漏洞
# 检查依赖安全性
# 检查敏感信息泄露
```

**检查点**：
- [ ] 安全扫描已完成（如适用）
- [ ] 漏洞已记录
- [ ] 修复建议已列出

### 第四步：自动测试

运行自动测试：

```bash
# 运行单元测试
# 运行集成测试
# 验证功能
```

**检查点**：
- [ ] 测试已运行（如适用）
- [ ] 测试结果已记录
- [ ] 失败的测试已列出

### 第五步：修复循环（验证闭环）

如果发现任何问题，进入**完整的修复验证循环**：

**修复验证流程**（强制执行）：

1. **识别问题**：列出所有发现的问题
2. **优先级排序**：确定修复优先级
3. **修复前分析**：
   - 问题模式识别（这个问题属于什么类型？）
   - 类似问题搜索（项目中是否有相同模式？）
   - 影响范围评估（修复会影响哪些功能？）
4. **执行修复**（dev-coder）：
   - 修复所有相关问题（不只是单个问题）
   - 确保修复的一致性
5. **修复后验证**（security-tester）：
   - ✅ 确认原始问题已修复
   - ✅ 确认所有类似问题已修复
   - ✅ 确认未引入新问题（回归检查）
   - ✅ 确认未破坏现有功能
6. **重新检查**：重新运行质量检查
7. **最多 3 次修复尝试**

**检查点**：
- [ ] 问题已识别
- [ ] 修复优先级已确定
- [ ] 问题模式已分析
- [ ] 类似问题已搜索
- [ ] 影响范围已评估
- [ ] 所有问题已修复（不是只修复单个问题）
- [ ] 修复后验证已完成（security-tester 验证）
- [ ] 回归检查已完成
- [ ] 重新检查通过
- [ ] （或达到最大尝试次数）

**禁止行为**：
- ❌ 只修复单个问题，不检查类似问题
- ❌ 修复后不验证是否引入新问题
- ❌ 修复后不检查是否破坏现有功能
- ❌ 发现新问题但不管

### 第六步：记录 Quality Gate 结果

将质量检查结果记录到状态文件：

```json
{
  "stage": "quality_gate",
  "static_analysis": {"status": "passed|failed", "issues": 0},
  "security_scan": {"status": "passed|failed", "vulnerabilities": 0},
  "tests": {"status": "passed|failed", "total": 10, "passed": 10, "failed": 0},
  "fix_attempts": 0,
  "result": "passed|failed"
}
```

**检查点**：
- [ ] 质量检查结果已记录
- [ ] 状态文件已更新
- [ ] 日志已记录

## 模式差异

| 特性 | 标准模式 | 完整模式 |
|------|----------|----------|
| 静态分析 | 可选 | 必须 |
| 安全扫描 | 可选 | 必须 |
| 自动测试 | 可选 | 必须 |
| 修复循环 | 建议修复 | 最多 3 次尝试 |
| 失败处理 | 记录并继续 | 必须通过才能继续 |

## 质量检查项目

### 1. 静态分析

**检查内容**：
- 语法正确性
- 代码风格
- 命名规范
- 代码结构
- 复杂度分析

**工具**：
- 语言特定的 linter
- 代码格式化工具
- 复杂度分析工具

### 2. 安全扫描

**检查内容**：
- 已知漏洞
- 依赖安全性
- 敏感信息泄露
- 权限问题
- 输入验证

**工具**：
- 安全扫描工具
- 依赖检查工具
- 密钥扫描工具

### 3. 自动测试

**检查内容**：
- 单元测试
- 集成测试
- 功能验证
- 边界测试
- 错误处理

**工具**：
- 测试框架
- 测试运行器
- 覆盖率工具

## 失败处理

### 质量检查失败

**标准模式**：
- 记录失败
- 给出修复建议
- 可以选择继续

**完整模式**：
- 记录失败到状态文件和日志
- 启动修复循环
- 最多 3 次修复尝试
- 如果 3 次后仍失败，询问用户是否继续

### 修复循环

**修复流程**：
1. 识别问题
2. 生成修复方案
3. 启动 dev-coder 修复问题
4. 重新运行质量检查
5. 如果仍然失败，重复（最多 3 次）

**修复记录**：
```json
{
  "attempt": 1,
  "issue": "问题描述",
  "fix": "修复方案",
  "result": "success|failure"
}
```

## 完成标志

当以下条件满足时，本阶段完成：

### 完整模式
- [ ] 所有质量检查已完成
- [ ] 所有问题已修复（或达到最大尝试次数）
- [ ] 质量检查结果已记录
- [ ] 状态文件已更新
- [ ] 日志已记录

### 标准模式
- [ ] 质量检查已完成（或跳过）
- [ ] 问题已列出（如有）
- [ ] （可选）状态文件已更新

## 超时处理

- **单次检查超时**：60 秒
- **修复循环超时**：根据问题数量
- **整体超时**：10 分钟
- **超时后操作**：记录超时，询问用户是否继续
