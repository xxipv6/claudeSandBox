# 代码审查

## 审查维度

### 1. 功能性

- [ ] 代码实现了预期功能
- [ ] 边界情况处理正确
- [ ] 错误处理完善
- [ ] 逻辑清晰易懂

### 2. 安全性

- [ ] 输入验证
- [ ] 输出编码
- [ ] 认证/授权
- [ ] 敏感数据保护
- [ ] 依赖安全

### 3. 性能

- [ ] 算法复杂度合理
- [ ] 无明显性能瓶颈
- [ ] 资源使用合理
- [ ] 缓存策略得当

### 4. 可读性

- [ ] 命名清晰
- [ ] 代码组织良好
- [ ] 注释适当
- [ ] 遵循项目规范

### 5. 可维护性

- [ ] 模块化设计
- [ ] 耦合度低
- [ ] 易于测试
- [ ] 易于扩展

### 6. 测试

- [ ] 有测试覆盖
- [ ] 测试用例充分
- [ ] 边界测试
- [ ] 错误测试

## 审查流程

### 第一步：理解上下文

1. **了解变更目的**
   - 为什么做这个变更？
   - 解决什么问题？

2. **理解变更范围**
   - 哪些文件被修改？
   - 影响哪些模块？

3. **查看相关文档**
   - 需求文档
   - 设计文档
   - API 文档

### 第二步：详细审查

#### 代码质量检查

```bash
# 使用工具辅助
- ESLint / Pylint / golangci-lint
- 静态分析工具
- 类型检查器
```

#### 安全检查

```bash
# 加载安全技能
读取：.claude/skills/security/vuln-patterns/
读取：.claude/rules/security.md

# 检查项
- 硬编码密钥
- SQL 注入
- XSS 漏洞
- CSRF 保护
- 输入验证
```

#### 最佳实践检查

- 设计模式使用
- SOLID 原则
- DRY 原则
- 错误处理模式

### 第三步：提供反馈

**反馈原则**：
- 具体明确（指出具体行和问题）
- 解释原因（为什么这样更好）
- 提供建议（如何改进）
- 区分优先级（必须修复 vs 建议改进）

**反馈模板**：

```
### 问题：[简短描述]

**位置**：`file:line`
**严重性**：[必须修复 / 建议改进 / 可选]

**问题描述**：
[具体说明问题]

**建议**：
[如何改进]

**示例**：
```diff
- old code
+ new code
```
```

## 使用示例

```bash
# 审查所有变更
/code-review

# 审查特定文件
/code-review src/auth/login.js

# 审查特定分支
/code-review --branch feature/login

# 生成审查报告
/code-review --report
```

## 输出格式

```
## 代码审查报告

### 变更概览
- 文件数：X
- 新增行：Y
- 删除行：Z
- 变更目的：[简述]

### 发现的问题

#### 必须修复 (2)
1. [安全] SQL 注入风险 - `api/users.js:45`
   - 问题：使用字符串拼接构建 SQL
   - 建议：使用参数化查询
   - 示例：
     ```diff
     - const query = `SELECT * FROM users WHERE id = ${id}`;
     + const query = 'SELECT * FROM users WHERE id = ?';
     + db.query(query, [id]);
     ```

2. [功能] 空指针异常 - `auth/service.js:78`
   - 问题：user 可能未定义
   - 建议：添加空值检查

#### 建议改进 (3)
1. [性能] 可以使用缓存 - `data/fetch.js:23`
2. [可读性] 函数过长，建议拆分 - `utils/formatter.js:56`
3. [测试] 缺少边界测试 - `tests/auth.test.js`

### 总结
- 整体评价：[良好 / 需要改进 / 需要重构]
- 主要问题：[总结]
- 建议行动：[下一步做什么]
```

## 审查技巧

### 快速扫描

1. 先看大的变更
2. 关注关键路径
3. 检查安全敏感代码
4. 验证错误处理

### 深度审查

1. 逐行阅读
2. 理解每个函数
3. 追踪数据流
4. 验证边界情况

### 工具辅助

```bash
# 自动化检查
npm run lint
npm run test
npm run audit

# 代码指标
npm run complexity  # 复杂度分析
npm run coverage   # 测试覆盖率
```

## 注意事项

- 保持建设性态度
- 解释"为什么"而非只是"是什么"
- 区分风格问题和实质问题
- 认可好的代码实践
- 提供学习资源
- 讨论而非命令
