# Tooling as Perspectives

工具不是能力，工具是观察角度。

**这是跨 agent 共享的工具认知库**。

---

## 核心原则

- **工具输出 ≠ 事实**：工具只报告它看到的东西
- **工具只能看到它设计要看的东西**：每个工具都有盲区
- **必须用其他视角验证**：单一工具的结论需要其他工具/视角验证
- **工具组合才能覆盖完整的攻击面**

---

## 视角类型

### 静态视角（Static Perspective）
**观察内容**：结构、关系、潜在路径

**典型工具**：
- **代码审计工具**（Semgrep, CodeQL, SonarQube）
  - 能看到：代码模式、已知漏洞签名、不安全的函数调用
  - 看不到：运行时行为、数据流值的范围、业务逻辑错误

- **依赖分析工具**（npm audit, pip-audit, Snyk）
  - 能看到：已知漏洞的依赖版本
  - 看不到：未公开的漏洞、依赖的实际使用情况

- **配置审计工具**（Kubernetes audit, Cloud custodian）
  - 能看到：配置错误、不安全的设置
  - 看不到：运行时的配置变更、环境变量的实际值

**局限性**：
- 无法理解业务逻辑
- 无法发现"合法操作的非法组合"
- 对未知漏洞模式无效

### 动态视角（Dynamic Perspective）
**观察内容**：实际执行、条件触发、状态变化

**典型工具**：
- **Fuzzing 工具**（AFL++, libFuzzer, honggfuzz）
  - 能看到：崩溃、异常、未处理的输入
  - 看不到：未触发的代码路径、业务逻辑漏洞

- **动态分析工具**（strace, ltrace, Frida, eBPF）
  - 能看到：系统调用、函数调用、运行时行为
  - 看不到：不发生的调用、潜在的调用路径

- **Web 扫描器**（Burp Suite, OWASP ZAP, sqlmap）
  - 能看到：HTTP 请求/响应、已知漏洞特征
  - 看不到：业务逻辑漏洞、需要多步骤的攻击

**局限性**：
- 覆盖率取决于输入生成
- 对业务逻辑理解有限
- 可能触发告警但无法确认利用性

### 覆盖视角（Coverage Perspective）
**观察内容**：输入覆盖、路径覆盖、状态覆盖

**典型工具**：
- **代码覆盖率工具**（gcov, llvm-cov, Istanbul）
  - 能看到：哪些代码被执行了
  - 看不到：哪些状态被覆盖了

- **状态覆盖率工具**（针对特定系统）
  - 能看到：哪些状态被访问了
  - 看不到：状态之间的转换是否完整

**局限性**：
- 状态覆盖率通常缺失
- 覆盖率 ≠ 安全性

### 人工视角（Human Perspective）
**观察内容**：业务逻辑、攻击意图、创造性利用

**优势**：
- 理解业务逻辑
- 发现"合法操作的非法组合"
- 创造性攻击路径构建

**局限性**：
- 慢、易出错
- 难以大规模覆盖

## 工具组合策略

### 策略 1：三角验证（Triangulation）
**原理**：用 3 个不同视角的工具验证同一问题

**示例**：
- 静态工具报告"SQL 注入风险"
- 动态工具确认"可触发"
- 人工验证"可利用"

### 策略 2：互补覆盖（Complementary Coverage）
**原理**：用不同工具覆盖不同盲区

**示例**：
- 静态工具：覆盖代码模式
- 动态工具：覆盖运行时行为
- 人工分析：覆盖业务逻辑

### 策略 3：层次递进（Layered Progression）
**原理**：从快速工具开始，逐步深入

**示例**：
1. 自动化扫描：发现明显问题
2. 针对性 fuzz：深入特定区域
3. 人工审计：理解业务逻辑
4. 手动利用：验证可利用性

## 必须自问的问题

### 使用工具前
- 这个工具能看到什么？
- 这个工具看不到什么？
- 这个工具的误报率如何？
- 这个工具的漏报率如何？

### 分析工具输出时
- 这个"发现"是工具的误报吗？
- 工具是否被"成功输出"误导？
- 是否需要其他工具验证？
- 是否需要人工验证？

### 分析工具阴性结果时
- 工具没发现问题 = 没问题吗？
- 工具是否覆盖了相关代码/路径？
- 是否需要换一个工具？

## 常见工具及其盲区

### Web 安全工具

| 工具 | 能看到 | 看不到 |
|------|--------|--------|
| Burp Suite | HTTP 流量、已知漏洞 | 业务逻辑漏洞、多步攻击 |
| sqlmap | SQL 注入点 | 其他类型漏洞、业务逻辑 |
| OWASP ZAP | 常见 Web 漏洞 | 业务逻辑、自定义逻辑 |
| Nessus | 已知 CVE、配置错误 | 业务逻辑、0-day |

### 二进制安全工具

| 工具 | 能看到 | 看不到 |
|------|--------|--------|
| IDA Pro | 反汇编代码、控制流 | 运行时行为、数据值 |
| Ghidra | 反编译代码 | 运行时行为、复杂逻辑 |
| AFL++ | 崩溃、异常 | 未崩溃路径、逻辑漏洞 |
| Frida | 运行时行为 | 不触发的代码路径 |

### 代码审计工具

| 工具 | 能看到 | 看不到 |
|------|--------|--------|
| Semgrep | 代码模式 | 业务逻辑、运行时行为 |
| CodeQL | 数据流、控制流 | 业务逻辑、动态值范围 |
| SonarQube | 代码质量问题 | 安全漏洞的利用性 |

### 动态分析工具

| 工具 | 能看到 | 看不到 |
|------|--------|--------|
| strace | 系统调用 | 用户态行为、业务逻辑 |
| ltrace | 库函数调用 | 不调用的函数、业务逻辑 |
| eBPF | 内核/系统行为 | 用户态逻辑 |

## 工具选择的决策树

```
问题类型？
├─ Web 应用
│  ├─ 快速扫描 → Burp Suite, OWASP ZAP
│  ├─ SQL 注入 → sqlmap
│  ├─ 业务逻辑 → 人工分析
│  └─ 全面审计 → 静态 + 动态 + 人工
├─ 二进制
│  ├─ 逆向 → IDA Pro, Ghidra
│  ├─ Fuzz → AFL++, libFuzzer
│  ├─ 运行时 → Frida, eBPF
│  └─ 全面审计 → 静态 + Fuzz + 人工
├─ 协议
│  ├─ 抓包 → Wireshark
│  ├─ Fuzz → boofuzz, SPIKE
│  └─ 逻辑分析 → 人工
└─ 代码审计
   ├─ 快速扫描 → Semgrep, SonarQube
   ├─ 深度分析 → CodeQL
   └─ 业务逻辑 → 人工
```

## 工具使用的最佳实践

### 1. 永远不要相信单一工具
- 每个工具都有盲区
- 必须用多个工具交叉验证
- 工具阴性 ≠ 安全

### 2. 理解工具的工作原理
- 工具如何检测问题？
- 工具的检测规则是什么？
- 工具的误报模式是什么？

### 3. 验证工具的输出
- 误报是常态
- 必须人工验证每个"发现"
- 不要自动信任工具的判断

### 4. 工具组合才是王道
- 静态 + 动态
- 自动化 + 人工
- 多个不同视角的工具

### 5. 持续更新工具
- 工具的规则库会更新
- 新的漏洞模式需要新工具
- 工具本身可能有漏洞

## 各 Agent 的工具使用建议

### product-manager
- **工具需求**：业务流程分析工具
- **建议**：更多依赖人工分析，工具辅助

### backend-engineer
- **工具需求**：代码审计 + 动态分析
- **建议**：CodeQL + strace/ltrace + 人工

### frontend-engineer
- **工具需求**：浏览器 DevTools + 代理
- **建议**：Chrome DevTools + Burp Suite

### qa-engineer
- **工具需求**：覆盖率 + 失败注入
- **建议**：gcov + chaos engineering tools

### security-tester
- **工具需求**：全工具链
- **建议**：静态 + 动态 + Fuzz + 人工
