# Tooling as Perspectives

工具不是能力，工具是观察角度。

**这是跨 agent 共享的工具认知库**。

---

## 核心原则

- **工具输出 ≠ 事实**：工具只报告它看到的东西
- **工具只能看到它设计要看的东西**：每个工具都有盲区
- **必须用其他视角验证**：单一工具的结论需要其他工具/视角验证
- **工具组合才能覆盖完整的攻击面**
- **视角比工具重要**：理解观察角度比选择具体工具更关键

---

## 视角类型

### 静态视角（Static Perspective）
**观察内容**：结构、关系、潜在路径

**能看到**：
- 代码模式、已知漏洞签名、不安全的函数调用
- 配置错误、不安全的设置
- 已知漏洞的依赖版本

**看不到**：
- 运行时行为、数据流值的范围、业务逻辑错误
- 未触发的代码路径、业务逻辑漏洞
- 运行时的配置变更、环境变量的实际值

**局限性**：
- 无法理解业务逻辑
- 无法发现"合法操作的非法组合"
- 对未知漏洞模式无效

---

### 动态视角（Dynamic Perspective）
**观察内容**：实际执行、条件触发、状态变化

**能看到**：
- 崩溃、异常、未处理的输入
- 系统调用、函数调用、运行时行为
- HTTP 请求/响应、已知漏洞特征

**看不到**：
- 未触发的代码路径、业务逻辑漏洞
- 不发生的调用、潜在的调用路径
- 业务逻辑漏洞、需要多步骤的攻击

**局限性**：
- 覆盖率取决于输入生成
- 对业务逻辑理解有限
- 可能触发告警但无法确认利用性

---

### 覆盖视角（Coverage Perspective）
**观察内容**：输入覆盖、路径覆盖、状态覆盖

**能看到**：
- 哪些代码被执行了
- 哪些状态被访问了

**看不到**：
- 哪些状态之间的转换是否完整
- 状态覆盖率通常缺失
- 覆盖率 ≠ 安全性

**局限性**：
- 状态覆盖率通常缺失
- 覆盖率 ≠ 安全性

---

### 人工视角（Human Perspective）
**观察内容**：业务逻辑、攻击意图、创造性利用

**优势**：
- 理解业务逻辑
- 发现"合法操作的非法组合"
- 创造性攻击路径构建

**局限性**：
- 慢、易出错
- 难以大规模覆盖

---

## 分析层 Agent 工具视角

### task-planner 视角

**关注点**：任务拆解、依赖识别、优先级排序

**观察角度**：
- **结构视角**：任务的组成和层次
- **依赖视角**：任务之间的关系和约束
- **优先级视角**：任务的重要性和紧急性

**典型工具类别**：
- 项目管理工具：任务列表、依赖关系、优先级
- 依赖分析工具：显式依赖关系
- 可视化工具：任务关系图、甘特图

**盲区**：
- 无法理解业务上下文
- 无法识别未文档化的依赖
- 优先级判断可能不准确

---

### product-manager 视角

**关注点**：需求分析、业务逻辑、成功标准

**观察角度**：
- **需求视角**：用户的真实目标
- **价值视角**：业务价值和优先级
- **边界视角**：做什么、不做什么

**典型工具类别**：
- 需求管理工具：文档化的需求
- 原型工具：设计稿、交互流程
- 用户故事工具：用户场景、用例

**盲区**：
- 难以理解技术实现细节
- 可能遗漏边界场景
- 成功标准可能不够具体

---

### backend-engineer 视角

**关注点**：系统结构、接口契约、状态机

**观察角度**：
- **架构视角**：组件之间的关系
- **接口视角**：接口定义和数据模型
- **状态视角**：状态转换和流转

**典型工具类别**：
- 代码分析工具：代码结构、已知漏洞模式
- API 文档工具：接口定义、数据模型
- 数据库工具：数据模型、关系

**盲区**：
- 静态分析无法理解运行时行为
- 接口定义可能与实际实现不一致
- 难以发现"合法操作的非法组合"

---

### frontend-engineer 视角

**关注点**：输入面、交互路径、攻击面

**观察角度**：
- **输入视角**：用户可以输入什么
- **交互视角**：用户可以进行哪些操作
- **暴露视角**：前端暴露了什么能力

**典型工具类别**：
- 浏览器开发工具：DOM、网络请求、存储
- 爬虫工具：可访问的页面、可执行的交互
- 代理工具：拦截和修改请求

**盲区**：
- 只能看到前端暴露的内容
- 无法直接观察后端逻辑
- 可能遗漏通过 API 直接调用的攻击路径

**观察角度**：
- **失败视角**：什么情况下会失败
- **边界视角**：输入边界、状态边界

### security-tester 视角

**关注点**：攻击路径、权限绕过、漏洞

**观察角度**：
- **攻击者视角**：攻击者可以做什么
- **路径视角**：如何达成攻击目标
- **绕过视角**：如何绕过防御机制

**典型工具类别**：
- 动态安全测试工具：已知的漏洞特征、可利用的路径
- 静态代码分析工具：代码中的漏洞模式
- 威胁建模工具：系统化的威胁识别

**盲区**：
- 无法发现未知漏洞模式
- 可能产生大量误报
- 难以理解业务逻辑漏洞

## 执行层 Coder Agent 工具视角

### dev-coder 视角

**关注点**：代码实现、错误处理、安全编码

**观察角度**：
- **实现视角**：如何实现需求
- **质量视角**：代码质量和可维护性
- **安全视角**：防止常见漏洞

**典型工具类别**：
- IDE：代码结构、语法错误
- Linter：代码风格问题、常见错误
- 代码格式化工具：代码格式

**盲区**：
- 工具只能发现语法和风格问题
- 无法发现逻辑错误
- 无法替代人工审查

---

### script-coder 视角

**关注点**：PoC、Exploit、Fuzzer、扫描工具

**观察角度**：
- **利用视角**：如何利用已知漏洞
- **自动化视角**：如何自动化攻击流程
- **验证视角**：如何验证漏洞存在性

**典型工具类别**：
- 调试工具：分析目标程序行为
- 网络工具：拦截和分析网络流量
- Fuzzing 框架：自动化输入生成

**盲区**：
- 依赖公开的漏洞数据库
- 可能需要针对特定目标定制
- 误报率可能较高

---

## 支持层 Agent 工具视角

### ops-engineer 视角

**关注点**：环境配置、工具安装、系统调试

**观察角度**：
- **环境视角**：系统配置和状态
- **资源视角**：资源使用情况
- **问题视角**：系统问题的诊断

**典型工具类别**：
- 包管理器：已安装的包、版本
- 系统监控工具：系统资源使用情况
- 日志工具：系统日志和事件

**盲区**：
- 只能看到当前状态
- 难以诊断间歇性问题
- 需要结合其他工具才能定位根因

## 视角组合策略

### 策略 1：三角验证（Triangulation）
**原理**：用 3 个不同视角的工具验证同一问题

**示例**：
- 静态工具报告"风险"
- 动态工具确认"可触发"
- 人工验证"可利用"

### 策略 2：分层验证（Layered Verification）
**原理**：在不同层次验证问题

**示例**：
- 前端工程师：输入面分析
- 后端工程师：接口契约分析
- 安全工程师：攻击路径分析

### 策略 3：互补视角（Complementary Perspectives）
**原理**：用能互相补充盲区的工具

**示例**：
- 静态分析：静态代码分析
- 动态分析：动态运行时分析
- 人工审查：业务逻辑分析

## 工具选择原则

### 根据任务类型选择视角

| 任务类型 | 推荐视角 | 禁止视角 |
|---------|----------|----------|
| 状态机分析 | 架构视角、状态视角 | 无 |
| 边界分析 | 测试视角、输入视角 | 无 |
| 攻击路径分析 | 攻击者视角、绕过视角 | 无 |
| 代码实现 | 实现视角、质量视角 | 无 |
| 环境配置 | 环境视角、资源视角 | 无 |

### 根据系统类型选择视角

| 系统类型 | 推荐视角 | 注意事项 |
|---------|----------|----------|
| Web 应用 | 交互视角、输入视角 | 注意 CSRF、XSS |
| API | 接口视角、认证视角 | 注意认证、授权 |
| 二进制 | 逆向视角、调试视角 | 注意逆向限制 |
| 协议 | 流量视角、解析视角 | 注意协议解析 |

## 工具输出解读指南

### 静态分析工具输出
- **高置信度**：通常是真的，但需要验证
- **中置信度**：可能是真的，需要人工验证
- **低置信度**：需要验证才能确认

### 动态分析工具输出
- **崩溃**：确实有问题，但需要分析根因
- **异常**：可能是问题，也可能是正常行为
- **超时**：可能是性能问题，也可能是工具问题

### 人工审查输出
- **主观判断**：需要其他视角验证
- **经验判断**：记录推理过程
- **不确定性**：明确标记为假设

---

## 工具视角的局限性

### 通用局限性
1. **工具只能看到它设计要看的东西**
2. **工具输出不等于事实**
3. **单一工具无法覆盖所有角度**
4. **工具组合才能完整覆盖**

### 特定场景的局限性
1. **业务逻辑漏洞**：工具难以发现
2. **零日漏洞**：工具无法检测
3. **复合攻击**：需要多工具组合
4. **社会工程**：工具无法检测

### 如何弥补局限性
1. **多视角验证**：用不同工具验证同一问题
2. **人工分析**：补充工具无法覆盖的部分
3. **持续更新**：定期更新工具和规则
4. **记录盲区**：记录工具的已知盲区
