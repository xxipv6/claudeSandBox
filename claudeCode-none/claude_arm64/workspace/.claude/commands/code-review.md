# 代码审查

## 功能说明

并发进行前后端代码审查，检查代码质量、安全性和可维护性。

---

## 执行流程

### 第一步：识别代码类型

分析用户输入或 git diff，识别：

**后端代码特征**：
- API 路由、控制器
- 数据模型、数据库操作
- 业务逻辑、服务层
- 中间件、拦截器

**前端代码特征**：
- UI 组件、页面
- 状态管理
- 路由配置
- 样式文件

---

### 第二步：并发调用 Agents

**后端代码审查**：
```bash
调用：backend-engineer agent

Agent 职责：
- 使用 code-review skill 审查后端代码
- 检查 6 个维度：功能性、性能、可读性、可维护性、测试、安全性
- 生成后端审查报告
```

**前端代码审查**：
```bash
调用：frontend-engineer agent

Agent 职责：
- 使用 code-review skill 审查前端代码
- 检查 6 个维度：功能性、性能、可读性、可维护性、测试、安全性
- 生成前端审查报告
```

**并发执行**：
- 后端和前端审查同时进行
- 等待两个 agent 完成
- 汇总审查结果

---

### 第三步：汇总结果

```markdown
## 代码审查报告

### 变更概览
- 文件数：X
- 后端文件：Y
- 前端文件：Z
- 变更目的：[简述]

[backend-engineer agent 的输出]

#### 总结
- 整体评价：[良好/需要改进/需要重构]
- 主要问题：[总结]

### 前端审查结果

#### 发现的问题
[frontend-engineer agent 的输出]

#### 总结
- 整体评价：[良好/需要改进/需要重构]
- 主要问题：[总结]

- 优先修复：[列出的高优先级问题]

---

## 使用方式

### 审查所有变更

```bash
/code-review

# Claude 会：
# 1. 分析 git diff
# 2. 识别前后端代码
# 3. 并发调用 backend-engineer 和 frontend-engineer
# 4. 汇总审查报告
```

### 审查特定文件

```bash
# 后端文件
/code-review src/api/users.js

# 前端文件
/code-review src/components/Login.jsx

# 多个文件
/code-review src/auth/ src/components/
```

### 审查特定分支

```bash
/code-review --branch feature/login

# 等同于：
git diff main...feature/login | /code-review
```

## 输出格式

### 问题格式

```markdown
#### [必须修复 / 建议改进 / 可选] - [问题类型]

**位置**：`file:line`
**严重性**：[高 / 中 / 低]

**问题描述**：
[具体说明问题]

**建议**：
[如何改进]

**示例**：
```diff
- old code
+ new code
```
```

### 总结格式

```markdown
### 总结

**整体评价**：[良好 / 需要改进 / 需要重构]

**主要问题**：
1. [问题 1]
2. [问题 2]
3. [问题 3]

**建议行动**：
- [ ] 优先修复 [高优先级问题]
- [ ] 后续改进 [中优先级问题]
- [ ] 可选优化 [低优先级问题]

**学习资源**：
[相关文档或最佳实践链接]
```

## 工作流程图

```
用户输入
    ↓
识别代码类型
    ↓
    ├─→ 后端代码 ──→ backend-engineer agent ──┐
    │                                          ├─→ 汇总报告
    └─→ 前端代码 ──→ frontend-engineer agent ──┘
```

---

## 注意事项

1. **并发执行**：前后端审查同时进行，节省时间
2. **Agent 职责**：agent 负责实际审查，命令负责协调和汇总
3. **Skill 使用**：agent 使用 `code-review` skill 进行审查
4. **审查维度**：6 个维度全覆盖（功能性、性能、可读性、可维护性、测试、安全性）
