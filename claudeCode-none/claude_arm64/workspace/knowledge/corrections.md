**这是跨 agent 共享的学习库**，所有 agent 都应避免重复犯错。

## 预填充的常见错误模式

## [2025-01-01] 边界检查不一致

### 问题空间
边界

### 错误假设
前端检查了输入，后端不需要再检查。

### 正确模型
前端检查是用户体验优化，后端必须独立检查所有边界。

### 为什么会错
认知盲区：假设前端不可绕过。

### 以后如何避免
- 所有边界检查必须在后端强制执行
- 前端检查仅用于提前反馈，不替代后端检查
- API 设计必须假设调用方可以是恶意的

---

## [2025-01-02] 状态机绕过

### 问题空间
状态、组合

### 错误假设
用户会按正常流程操作，不会跳过步骤。

### 正确模型
用户会尝试所有可能的操作组合，包括跳过步骤。

### 为什么会错
认知盲区：假设用户遵循设计意图。

### 以后如何避免
- 所有状态转换必须在后端强制验证
- 每个状态转换都要检查：当前状态是什么？能否到达目标状态？
- 禁止依赖前端控制状态转换

---

## [2025-01-03] 信任传递链

### 问题空间
信任、边界

### 错误假设
内部服务之间的调用可以相互信任。

### 正确模型
即使是内部服务，也要验证权限和身份。

### 为什么会错
假设错误：假设内部网络是安全的。

### 以后如何避免
- 所有跨服务调用必须传递和验证身份
- 零信任原则：每个请求都要验证
- 服务网格/mTLS 不能替代应用层验证

---

## [2025-01-04] TOCTOU 竞态

### 问题空间
时间、状态

### 错误假设
检查和执行之间不会有状态变化。

### 正确模型
检查和执行之间存在时间窗口，状态可能变化。

### 为什么会错
认知盲区：忽略了并发和时间窗口。

### 以后如何避免
- 检查和执行必须原子化
- 使用事务、锁、原子操作
- 设计时避免"检查后使用"模式

---

## [2025-01-05] 资源泄露

### 问题空间
资源、状态

### 错误假设
函数返回时会自动释放资源。

### 正确模型
只有显式释放或使用 RAII，资源才会释放。

### 为什么会错
工具误导：某些语言的自动释放机制不可靠。

### 以后如何避免
- 使用 context manager/RAII 模式
- 异常情况下也要释放资源（finally）
- 定期检查资源泄露（linter, monitoring）

---

## [2025-01-06] 批量操作绕过

### 问题空间
组合、边界

### 错误假设
单次操作的限制对批量操作也有效。

### 正确模型
批量操作可能绕过单次限制。

### 为什么会错
认知盲区：没有考虑操作组合。

### 以后如何避免
- 批量操作要有额外的配额检查
- 批量操作配额 ≤ 单次配额 × 批次大小
- 考虑速率限制（rate limiting）

---

## [2025-01-07] 时间窗口漏洞

### 问题空间
时间、状态

### 错误假设
验证码生成后立即使用，时间窗口很短所以安全。

### 正确模型
任何时间窗口都可被利用，需要限制使用次数。

### 为什么会错
认知盲区：忽略了攻击者可以自动化利用。

### 以后如何避免
- 验证码/临时凭证限制使用次数
- 使用后立即失效
- 缩短时间窗口（如 60 秒）

---

## [2025-01-08] 信息泄露

### 问题空间
边界、信任

### 错误假设
错误消息只是给开发者看的，不会泄露敏感信息。

### 正确模型
所有错误消息都会被用户看到，可能泄露系统细节。

### 为什么会错
认知盲区：忽略了错误消息的攻击面。

### 以后如何避免
- 生产环境返回通用错误消息
- 详细错误只记录到日志
- 错误码设计：对用户友好，对开发者有用

---

## [2025-01-09] 权限提升

### 问题空间
边界、信任

### 错误假设
用户无法修改自己的权限字段。

### 正确模型
用户可以操纵请求，修改任何字段。

### 为什么会错
信任假设错误：信任前端传递的数据。

### 以后如何避免
- 后端验证：用户不能修改自己无权修改的字段
- 使用白名单：只接受允许修改的字段
- 权限检查：每个操作都要检查权限

---

## [2025-01-10] 日志注入

### 问题空间
输入、信任

### 错误假设
日志只是用于调试，不需要清理输入。

### 正确模型
日志可能被查看、分析、导出，注入可能导致安全问题。

### 为什么会错
认知盲区：忽略了日志的后续用途。

### 以后如何避免
- 记录日志前清理/转义用户输入
- 使用结构化日志（JSON）
- 限制日志的访问权限

---

## [2025-01-11] 并发状态转换

### 问题空间
时间、状态、组合

### 错误假设
同一时间只有一个操作会修改状态。

### 正确模型
多个操作可能并发修改状态，导致不一致。

### 为什么会错
认知盲区：忽略了并发场景。

### 以后如何避免
- 使用锁/事务保证原子性
- 状态转换必须是幂等的
- 考虑使用乐观锁/版本号

---

## [2025-01-12] 默认配置不安全

### 问题空间
信任、边界

### 错误假设
用户会修改默认配置为安全的值。

### 正确模型
大多数用户不会修改默认配置。

### 为什么会错
认知盲区：假设用户会遵循最佳实践。

### 以后如何避免
- 默认配置必须安全
- 敏感功能默认关闭
- 明确文档化每个配置的风险

---

## [2025-01-13] 依赖漏洞

### 问题空间
信任、组合

### 错误假设
依赖库是安全的，因为有很多人在用。

### 正确模型
依赖库可能有漏洞，即使很流行。

### 为什么会错
信任假设错误：信任流行的依赖。

### 以后如何避免
- 定期审计依赖（npm audit, pip-audit）
- 使用依赖锁定文件
- 及时更新依赖到安全版本

---

## [2025-01-14] 环境变量泄露

### 问题空间
边界、信任

### 错误假设
环境变量只有服务器能访问。

### 正确模型
环境变量可能通过调试接口、错误消息、日志泄露。

### 为什么会错
认知盲区：忽略了环境变量的泄露路径。

### 以后如何避免
- 敏感信息使用密钥管理服务（KMS）
- 限制环境变量的访问权限
- 不在日志中记录环境变量

---

## [2025-01-15] JSON 注入

### 问题空间
输入、信任

### 错误假设
JSON 解析器会自动处理特殊字符。

### 正确模型
JSON 解析器可能执行恶意内容（如 prototype pollution）。

### 为什么会错
工具误导：假设 JSON 解析器是安全的。

### 以后如何避免
- 使用安全的 JSON 解析器
- 验证 JSON 结构和内容
- 避免 eval/Function 执行 JSON

## 通用规则总结

### 边界相关
1. 永远不要信任前端验证
2. 后端必须独立验证所有边界
3. 批量操作要有额外检查
4. 跨边界调用必须重新验证

### 状态相关
1. 所有状态转换必须后端强制验证
2. 检查和执行必须原子化
3. 并发操作要考虑竞态条件
4. 状态不一致是常见漏洞源

### 信任相关
1. 零信任原则：验证所有输入
2. 内部调用也要验证权限
3. 默认配置必须安全
4. 不信任任何"隐式安全"假设

### 时间相关
1. 任何时间窗口都可被利用
2. 限制验证码/临时凭证的使用次数
3. TOCTOU 是常见漏洞模式
4. 并发操作要保证原子性

### 资源相关
1. 使用 RAII/context manager 管理资源
2. 异常情况也要释放资源
3. 定期检查资源泄露
4. 资源配额要考虑批量操作

### 组合相关
1. 单独安全的操作组合后可能不安全
2. 考虑所有可能的操作组合
3. 业务逻辑漏洞最难发现
4. 人工分析是必须的

## 如何使用这个文件

### 对所有 Agent
1. **任务开始前**：快速浏览相关错误模式
2. **分析/编写时**：对照错误模式，避免重复犯错
3. **发现新错误**：按格式记录到这个文件

### 对 Orchestrator
- 在并行调度 subagents 前，提醒它们参考这个文件
- 在合并结果时，检查是否有重复的错误模式
- 将新发现的错误模式记录到这个文件

## 记录新错误的决策树

```
发现错误？
├─ 这个错误是第一次遇到？
│  ├─ 是 → 记录到这个文件
│  └─ 否 → 检查是否已有记录
│     ├─ 有 → 更新现有记录
│     └─ 没有 → 新增记录
└─ 这个错误是否值得记录？
   ├─ 是 → 记录
   └─ 否 → 忽略（如拼写错误、临时问题）
```

---

## 记录新错误的价值判断标准

**值得记录的错误**：
- 跨项目通用的问题
- 模式化的错误（不是一次性错误）
- 可预防的错误
- 有学习价值的错误

**不值得记录的错误**：
- 项目特定的临时问题
- 一次性错误（如拼写错误）
- 环境相关的问题（如临时网络故障）
- 已有文档覆盖的问题
