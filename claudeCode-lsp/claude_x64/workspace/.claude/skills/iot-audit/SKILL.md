# IoT 安全审计（自动识别版）

## Trigger

### TRIGGER WHEN
- 用户请求审计路由器、摄像头、智能家居、工控或其他嵌入式设备
- 检测到 `.bin` / `.img` / `.trx` / `.fw` / rootfs / squashfs / cramfs / jffs2 等固件资产
- 需要分析 bootloader、kernel、rootfs、升级机制、默认凭证、后门或持久化风险
- 需要在 “仅固件 / 仅源码 / 固件+源码” 三类 IoT 资产间自动判断路径
- 需要从远程、局域网、物理接触三个攻击者模型审计设备

### DO NOT TRIGGER WHEN
- 只是普通 Web 应用或后台系统白盒审计
- 只是某个单独二进制文件的逆向分析
- 只是普通源码安全审计，没有设备/固件/嵌入式上下文
- 只需要 `.jar` / `.class` 反编译或一次性 PoC 复现

### USE WITH
- 固件中的单个 native 组件深挖可配合 `binary-reverse`
- 需要 Web 管理后台白盒时可配合 `web-whitebox-audit`
- 需要验证设备漏洞可利用性时可配合 `poc-exploit`

### EXAMPLE PROMPTS
- “审计这个路由器固件升级链有没有签名绕过”
- “分析这个摄像头 rootfs 里的默认凭证和后门”
- “这个嵌入式设备源码+固件一起看，给我完整审计路径”
- “检查这个 IoT 设备有没有长期持久化风险”

---

## 设计目标

在未知交付形态下（只有固件 / 只有源码 / 固件+源码），
自动识别可用资产 → 选择正确审计路径 → 统一风险建模与输出。

---

## 一、自动识别阶段（必须先做）

这是整个 Skill 的入口阶段，决定后续走哪条路。

### 1️⃣ 资产形态识别规则

**检测到以下任一项 → 认为"有源代码"**：
- Makefile / CMakeLists.txt / Kconfig
- src/ / components/ / packages/
- 明显的 C/C++/Rust/Go 源文件

**检测到以下任一项 → 认为"有固件"**：
- .bin / .img / .trx / .fw
- squashfs / cramfs / jffs2
- bootloader / kernel / rootfs

### 2️⃣ 自动分类结果（只会落在一种）

- **Type A：仅固件**
- **Type B：仅源代码**
- **Type C：固件 + 源代码（混合）**

后续流程严格按类型分支执行，但输出模型统一。

---

## 二、统一审计主线（不随类型变化）

无论哪种形态，审计目标不变，只变"证据来源"。

**四条主线（强制贯穿）**：

1. **入口主线**：设备暴露了哪些可触发点
2. **权限主线**：设备内部"谁能做什么"
3. **状态主线**：设备状态是否可被非法推进
4. **持久化主线**：攻击是否能长期存在

---

## 三、Type A：仅固件审计路径

### 核心方法
- 固件解包
- 启动链还原
- 二进制逆向
- 配置与脚本分析

### 关键步骤
1. **rootfs 枚举**（/bin /sbin /etc / www）
2. **启动脚本**（rcS / systemd）
3. **网络服务识别**
4. **设备端二进制逆向**（管理服务 / 升级程序）

### 高危关注
- 默认 root / Telnet / UART
- 局域网接口无认证
- 升级验签绕过
- 隐藏命令 / 后门

---

## 四、Type B：仅源代码审计路径

### 核心方法
- 构建系统分析
- 入口函数枚举
- 权限与状态机代码审计
- 升级链源码审计

### 关键步骤
1. **启动链源码**（init / 守护进程）
2. **网络 / 串口 / 文件入口函数**
3. **认证与权限校验点**
4. **状态字段与状态迁移逻辑**

### 高危关注
- Debug 宏控制安全逻辑
- token 校验结果未使用
- 状态与权限解耦
- system/popen 参数拼接

---

## 五、Type C：固件 + 源代码（最强路径）

### 核心方法
- **源码为主，固件为证据**
- 源码 → 固件映射验证

### 关键步骤
1. **源码中定位**：
   - 网络服务
   - 升级逻辑
   - 权限判断

2. **固件中验证**：
   - 是否真的编译进 release
   - 是否存在额外隐藏逻辑

3. **对比**：
   - 源码 vs 实际二进制行为

### 高危关注
- 源码已修复但固件未更新
- 条件编译导致 release 行为不同
- 固件中存在源码未体现的后门

---

## 六、统一的"入口 → 权限 → 状态 → 副作用"模型

无论 Type A/B/C，每个发现都必须映射到这个模型：

```
入口点
  ↓
认证 / 权限校验
  ↓
状态判断
  ↓
危险副作用（执行 / 升级 / 持久化）
```

**任何一层缺失，都是漏洞候选。**

---

## 七、自动化攻击路径建模（统一）

### 攻击者能力模型（固定三类）
1. **远程未认证** - 从互联网直接攻击
2. **局域网攻击者** - 同一局域网内
3. **物理接触攻击者** - 有设备物理访问权限

### 构造方式
- 合法接口调用
- 合法参数
- 非预期结果

---

## 八、统一漏洞输出格式

每个漏洞包含：

- **设备型号 / 固件版本**
- **资产类型**（仅固件 / 仅源码 / 混合）
- **入口点**
- **执行路径**
- **漏洞类型**（越权 / 后门 / 升级链 / 硬编码凭证）
- **影响**（远程未认证 / 局域网 / 物理）
- **修复建议**（设计级）

---

## 常见 IoT 漏洞模式

### 硬编码凭证
- 默认 root / admin 密码
- Telnet / SSH 默认开启
- API 硬编码 token

### 认证绕过
- 升级验签缺失 / 弱校验
- 调试接口未关闭
- 隐藏后门命令

### 网络服务暴露
- 无认证的 Web 管理界面
- 未加密的 Telnet / HTTP
- UART / JTAG 暴露调试接口

### 固件更新漏洞
- 升级包签名验证缺失
- 差分更新无完整性校验
- 版本回滚机制缺失
