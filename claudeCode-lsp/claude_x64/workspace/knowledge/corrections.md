# 错误学习库（Corrections）

## 概述

这个文件记录了系统中常见的错误模式及其正确的修复方式，帮助 agents 避免重复犯错。

---

## 错误模式 001：不完整的 Bug 修复

### 错误描述

当用户指出一个 bug 时，系统只修复了指出的具体问题，而没有：
1. 检查项目中是否有类似的相同问题
2. 验证修复是否引入了新的 bug（回归缺陷）
3. 验证修复是否破坏了现有功能
4. 报告发现的所有相关问题

### 正确的修复流程

**完整的验证闭环**：

```
用户指出 bug
    ↓
1. 问题模式识别
   - 这个问题属于什么类型？
   - 这个问题是否可能在其他地方也存在？
   - 是否有相同的代码模式？
    ↓
2. 搜索类似问题
   - 在当前文件中搜索相同的模式
   - 在项目其他文件中搜索相同模式
   - 检查相关的函数、组件、模块
    ↓
3. 影响评估
   - 修复会影响哪些功能？
   - 是否会破坏现有逻辑？（回归风险）
   - 是否需要更新相关代码？
    ↓
4. 完整修复
   - 修复用户指出的具体问题
   - 修复所有类似问题（举一反三）
   - 更新相关代码和文档
    ↓
5. 修复后验证
   - ✅ 确认原始问题已修复
   - ✅ 确认所有类似问题已修复
   - ✅ 确认未引入新 bug（回归检查）
   - ✅ 确认未破坏现有功能
    ↓
6. 报告所有发现
   - 列出修复的所有问题
   - 报告发现的任何额外问题
   - 提供改进建议
```

### 错误示例

#### 示例 1：只修复指出的 bug

**用户说**："第 15 行的变量名不对"

**❌ 错误做法**：
```jsx
// 只改第 15 行
const userInfo = getUserInfo();  // 改为 userData
```

**✅ 正确做法**：
1. 读取完整文件
2. 分析第 15 行的上下文
3. 检查整个文件，发现 12 处使用了 `userInfo`
4. 检查项目中其他文件，发现 8 个文件也使用了相同模式
5. 修复所有文件中的命名问题（20 处）
6. 验证没有破坏功能
7. 报告所有修复

#### 示例 2：修复后不验证回归

**用户说**："这个 API 调用没有错误处理"

**❌ 错误做法**：
```jsx
// 只给这个 API 调用加 try-catch
try {
  const data = await fetchData();
} catch (error) {
  console.error(error);
}
// 不管了，没有验证是否引入新问题
```

**✅ 正确做法**：
1. 识别模式：所有异步操作都应该有错误处理
2. 搜索整个项目，发现 35 个异步调用缺少错误处理
3. 修复所有异步调用的错误处理（35 处）
4. 添加全局错误处理中间件
5. **验证修复不会破坏现有功能**
6. **检查是否引入了新的 bug**
7. 报告所有修复和发现

#### 示例 3：引入新 bug 后不管

**用户说**："这个查询有 SQL 注入风险"

**❌ 错误做法**：
```python
# 只修复这一个查询
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
# 但修复过程中，不小心破坏了其他功能
# 不管了，直接结束
```

**✅ 正确做法**：
1. 识别模式：所有直接拼接 SQL 的地方都有注入风险
2. 搜索整个项目，发现 18 处字符串拼接 SQL
3. 修复所有 SQL 注入风险（18 处）
4. **验证修复不会破坏功能和性能**
5. **如果发现引入了新 bug，立即修复**
6. **运行测试确保没有回归**
7. 报告所有修复和验证结果

### 术语定义

**局部修复（Local Fix）**：只修表面，不查根本原因，不检查类似问题 ❌

**修复范围过窄（Narrow Fix Scope）**：缺乏全局视野，只修复单个问题 ❌

**缺乏关联分析（Lack of Impact Analysis）**：没有分析相关问题，没有检查影响范围 ❌

**回归缺陷（Regression Bug）**：修复破坏了其他功能或引入了新 bug ❌

**副作用（Side Effect）**：修复引发的意外问题 ❌

**修复引入的缺陷（Fix-introduced Bug）**：修复过程中产生的新 bug ❌

**完整修复（Complete Fix）**：不仅修复表面问题，还要找到根本原因，修复所有类似问题 ✅

**验证闭环（Verification Loop）**：修复 → 验证 → 回归测试 → 报告 ✅

### 关键原则

1. **不要只修复表面问题** - 要找到根本原因
2. **不要只修复一处** - 要修复所有类似问题
3. **不要破坏现有功能** - 要评估修复影响
4. **不要假设没问题** - 要验证修复结果
5. **不要引入新 bug** - 修复后要验证回归
6. **不要遗漏报告** - 任何发现的问题都要报告

## 错误模式 002：只读计划不执行（完整模式）

### 错误描述

在完整模式（full-mode.md）的第二步中，用户期望的是"只读计划"（列出要做什么，标注参考文件），但系统可能误解为要执行这些文件。

### 正确理解

**第二步**：制定详细计划
- **目的**：列出计划，标注参考文件
- **操作**：只读，不执行
- **输出**：阶段列表，每个阶段标注参考文件路径

**第四步**：按顺序执行各阶段
- **目的**：实际执行各个阶段
- **操作**：读取阶段文件，按照指示执行

### 修复

在 full-mode.md 第二步中添加明确的警告：

```markdown
**⚠️ 说明**：此步骤仅列出计划并标注参考文件，**不执行任何操作**。实际执行在第四步进行。
```
