# 快速修复

## 何时使用

- Bug 修复
- 小改动
- 文档更新
- 配置调整
- 不需要规划的任务

## 快速流程

### 第一步：理解问题

1. **获取上下文**
   - 问题描述
   - 错误消息
   - 相关文件

2. **定位问题**
   - 搜索相关代码
   - 查看日志
   - 复现问题

### 第二步：实施修复

1. **直接修复**
   - 修改代码
   - 添加测试
   - 验证修复

2. **快速验证**
   - 本地测试
   - 手动验证
   - 确认修复

### 第三步：提交

1. **简单文档**
   - 更新注释
   - 更新文档（如需要）

2. **提交代码**
   ```bash
   git add .
   git commit -m "fix: ..."
   ```

## 使用示例

```bash
# 修复 bug
/quick "修复登录页面的验证错误"

# 更新配置
/quick "修改 API endpoint 为新的地址"

# 小功能
/quick "添加日志输出"

# 文档更新
/quick "更新 README 的安装说明"
```

## 输出

```
## 快速修复报告

### 问题描述
[用户描述的问题]

### 问题定位
- 位置：file:line
- 原因：[问题原因]

### 修复内容
- 修改：file:line
- 变更：[具体修改]

### 验证结果
✅ 修复已验证

### 提交
✅ 代码已提交
```

## 与 /full 的区别

| 维度 | /full | /quick |
|------|-------|--------|
| 规划 | ✅ 完整规划 | ❌ 跳过规划 |
| 分析 | ✅ 多维度分析 | ❌ 直接理解 |
| 实现 | ✅ 系统化实现 | ✅ 直接修复 |
| 验证 | ✅ 全面验证 | ✅ 快速验证 |
| 适合 | 复杂功能 | 简单修复 |

## 注意事项

- 适合简单任务
- 复杂任务用 /full
- 仍然要测试
- 注意安全性
