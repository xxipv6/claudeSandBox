# 安全审计

## 审计流程

### 第一步：信息收集

1. **识别目标范围**
   - 确认审计目标（应用、模块、API）
   - 确定审计范围和边界

2. **收集上下文**
   - 读取项目结构
   - 识别关键文件和入口点
   - 了解技术栈

### 第二步：漏洞分析

1. **加载安全技能**
   - 读取 `.claude/skills/security/vuln-analysis/`
   - 读取 `.claude/skills/security/vuln-patterns/`

2. **系统性扫描**
   - 检查常见漏洞模式（OWASP Top 10）
   - 验证安全规则（`.claude/rules/security.md`）
   - 识别潜在风险点

3. **深度分析**
   - 输入验证检查
   - 输出编码检查
   - 认证/授权分析
   - 加密使用检查

### 第三步：代码审查

1. **启动安全审查**
   - 调用 `security-tester` agent
   - 使用 `/secure-review` 命令

2. **检查清单**
   - ❌ 硬编码密钥
   - ❌ SQL 注入风险
   - ❌ XSS 漏洞
   - ❌ CSRF 保护
   - ❌ 不安全的随机数
   - ❌ 不安全的加密

### 第四步：威胁建模

1. **识别威胁**
   - 使用 STRIDE 方法论
   - 分析攻击面
   - 评估影响

2. **生成报告**
   - 漏洞清单
   - 风险评级（高/中/低）
   - 修复建议
   - 优先级排序

### 第五步：验证与建议

1. **PoC 验证**（可选）
   - 对关键漏洞生成 PoC
   - 使用 `/exploit-gen` 命令

2. **修复建议**
   - 提供具体修复方案
   - 安全编码最佳实践
   - 参考安全标准（CWE、OWASP）

## 使用示例

```bash
# 审计整个项目
/security-audit

# 审计特定模块
/security-audit src/auth/

# 审计 API
/security-audit api/
```

## 输出

审计完成后，生成安全审计报告：

```
## 安全审计报告

### 执行摘要
- 审计范围：...
- 发现漏洞：X 个高危，Y 个中危，Z 个低危

### 详细发现
1. [高危] SQL 注入 - src/auth/login.js:45
   - 描述：...
   - 修复建议：...

2. [中危] 缺少 CSRF 保护 - forms/contact.html
   - 描述：...
   - 修复建议：...

### 建议优先级
1. 立即修复：高危漏洞
2. 计划修复：中危漏洞
3. 最佳实践：低危问题
```

## 注意事项

- 安全审计需要明确的目标范围
- 对于生产环境，确保有授权
- 漏洞验证时注意安全（仅在测试环境）
- 遵循负责任的披露原则
