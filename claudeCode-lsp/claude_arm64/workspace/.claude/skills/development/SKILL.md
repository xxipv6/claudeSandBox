# 设计探索技能（Brainstorming）

## 何时激活

**必须使用的场景**（应主动（PROACTIVELY）使用）：
- 创建新功能
- 构建新组件
- 添加新行为
- 修改现有行为
- 设计新系统
- 重构现有代码

**核心原则**：
- ✅ **所有创造性工作都必须先经过设计探索**
- ❌ **禁止跳过设计直接实现**
- ❌ **即使看起来很简单，也必须经过此流程**

---

## 🎯 核心职责

1. **探索用户意图**：理解用户真正想要什么
2. **澄清需求**：明确目的、约束、成功标准
3. **探索方案**：提出 2-3 个方案并分析权衡
4. **呈现设计**：获得用户批准
5. **撰写设计文档**：保存到 docs/specs/

---

## 📋 执行流程

### 1. 探索项目上下文

```bash
# 检查文件结构
# 查看文档
# 查看最近提交
```

**目的**：理解当前项目状态

### 2. 判断是否需要视觉伴侣

**如果涉及视觉问题**，必须发送"视觉伴侣"邀请：

```
请启用视觉伴侣，我需要展示设计稿。
```

**何时需要视觉伴侣**：
- UI 布局
- 对比设计
- 架构图
- 流程图

**何时不需要**：
- 概念性问题
- 需求澄清

### 3. 提出澄清问题

**一次一个**，循序渐进地理解：

- 目的：为什么要做这个？
- 约束：有什么限制？
- 成功标准：如何判断成功？

**示例**：
```
Q1: 这个功能的主要目的是什么？
Q2: 目标用户是谁？
Q3: 有什么技术限制？
Q4: 如何衡量成功？
```

### 4. 提出 2-3 个方案

**每个方案包含**：
- 方案描述
- 优点
- 缺点
- 适用场景

**给出推荐**并解释原因

### 5. 呈现设计

**按复杂度分节**，每节后询问用户是否正确：

- 架构设计
- 组件结构
- 数据流
- 错误处理
- 测试策略

**示例**：
```markdown
## 架构设计

采用模块化架构，分为三个层次：
1. UI 层：负责用户交互
2. 业务逻辑层：处理数据
3. 数据访问层：与后端通信

是否正确？

## 组件结构

- MainLayout：主布局
- LoginForm：登录表单
- Dashboard：仪表板

是否正确？
```

### 6. 撰写设计文档

**路径**：`docs/specs/YYYY-MM-DD-<topic>-design.md`

**内容**：
```markdown
# [Topic] 设计文档

## 概述
[功能描述]

## 需求分析
[用户需求]

## 方案选择
[方案 1]
[方案 2]
[推荐方案及原因]

## 设计
### 架构
### 组件
### 数据流
### 错误处理
### 测试

## 实现计划
[下一步行动]
```

### 7. 规格审查循环

**调用 spec-document-reviewer 子代理**审查设计文档：

- 发现问题 → 修复 → 重新审查
- 最多 5 次
- 超过 5 次交给人类处理

### 8. 用户审查

**用户确认后才能继续**：

```
设计文档已保存到 docs/specs/YYYY-MM-DD-<topic>-design.md

请审阅设计文档，确认后我将进入实现阶段。
```

---

## 🚫 禁止行为

### ❌ 反模式

```
"这个太简单，不需要设计"
"直接写代码吧"
"先做个 prototype 看看"
```

**所有项目都必须经过设计流程**：
- ✅ 即使是待办清单
- ✅ 即使是单函数工具
- ✅ 即使是配置修改

### ❌ 禁止在 brainstorming 阶段

- 禁止调用实现类技能
- 禁止写代码
- 禁止 scaffold 项目
- 禁止任何实现行为

---

## ✅ 必须完成的检查清单

按顺序完成每一项：

- [ ] 探索项目上下文
- [ ] 判断是否需要视觉伴侣（如需要，发送邀请）
- [ ] 提出澄清问题（一次一个）
- [ ] 提出 2-3 个方案
- [ ] 呈现设计（分节，逐节确认）
- [ ] 撰写设计文档
- [ ] 规格审查循环（调用 spec-document-reviewer）
- [ ] 用户审查规格文档
- [ ] 进入实现阶段

## 🎯 设计原则

### 1. 隔离与清晰

**将系统拆成职责单一的小单元**

每个单元必须能回答：
- 它做什么？
- 如何使用？
- 它依赖什么？

**单元必须可独立理解与测试**

### 2. 文件大小

**文件过大通常意味着职责不清**

- 警惕：超过 300 行
- 危险：超过 500 行
- 考虑拆分

### 3. 在现有代码库中工作

- 先理解现有结构
- 遵循现有模式
- 如果现有代码影响当前工作，可提出针对性改进
- **不要提出与当前目标无关的重构**

## 🔄 完整流程图

```
探索项目上下文
    ↓
是否涉及视觉问题？
    ├─ 是 → 发送视觉伴侣邀请
    ↓
提出澄清问题（一次一个）
    ↓
提出 2-3 个方案（含权衡）
    ↓
呈现设计（分节，逐节确认）
    ↓
用户批准设计？
    ├─ 否 → 修订并重新呈现
    └─ 是 → 撰写设计文档
    ↓
规格审查循环（子代理审查）
    ↓
审查通过？
    ├─ 否 → 修复并重新审查
    └─ 是 → 用户审查规格文档
    ↓
用户批准规格文档？
    ├─ 否 → 修改并重新进入审查循环
    └─ 是 → 调用 planner agent 生成执行计划
```

## 📝 示例

### 场景：添加登录功能

```
# 探索项目上下文
[检查现有代码，发现没有认证]

# 澄清问题
Q: 需要支持哪些登录方式？
A: 邮箱密码 + 第三方登录（Google、GitHub）

Q: 是否需要多因素认证？
A: 暂时不需要

# 提出方案
方案 1：使用 JWT
  优点：无状态、易于扩展
  缺点：无法主动失效

方案 2：使用 Session
  优点：可控制、安全性高
  缺点：需要存储、扩展性差

推荐：方案 1（JWT）

# 呈现设计
[分节呈现，每节确认]

# 撰写设计文档
保存到 docs/specs/2026-03-16-login-design.md

# 规格审查
[调用 spec-document-reviewer]

# 用户审查
等待用户批准
```

## 🚀 进入实现阶段

**设计文档获得用户批准后**：

1. 调用 **planner agent** 生成执行计划
2. planner 指定"执行智能体：dev"
3. 自动调用 **dev agent** 开始实现
4. 如果需要测试，调用 **tdd-guide agent**

**注意**：brainstorming 之后唯一允许调用的是 planner，不能直接调用 dev 或其他实现 agent。

## 🎨 视觉伴侣使用指南

### 何时使用

**需要视觉呈现时**：
- UI 布局设计
- 对比不同设计
- 架构图
- 流程图
- 数据流图

### 使用方式

**发送独立邀请消息**：
```
请启用视觉伴侣，我需要展示设计稿。
```

**用户同意后**：
- 使用视觉伴侣展示 mockup
- 使用视觉伴侣展示图表
- 使用视觉伴侣展示布局

### 每个问题都要判断

不是所有问题都需要视觉伴侣：
- 概念性问题 → 文字
- 需求澄清 → 文字
- 设计展示 → 视觉伴侣

---

**记住**：所有创造性工作都必须先经过设计探索。这是确保项目成功的关键步骤。
