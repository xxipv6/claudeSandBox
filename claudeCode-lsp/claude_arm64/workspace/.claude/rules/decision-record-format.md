# Decision Record Format

## 触发条件

以下情况必须生成 Decision Record：

1. **初始研究**：开始新的研究项目
2. **路径变化**：改变研究方向或攻击面
3. **否决方向**：排除某个研究路径
4. **新攻击面**：发现新的攻击面
5. **启用多 Agent**：从 Single-Agent 切换到 Multi-Agent
6. **关键假设**：提出关键假设需要验证

---

- [理由 1]

---

## Decision ID 命名规则

格式：`YYYY-MM-DD-XXX`

- `YYYY-MM-DD`：决策日期
- `XXX`：当天决策序号（001, 002, 003...）

**示例**：
- `2026-03-25-001`：2026年3月25日的第1个决策
- `2026-03-25-002`：2026年3月25日的第2个决策

---

## 保存位置

```
xxx-research/docs/decisions/YYYY-MM-DD-decision-description.md
```

**示例**：
```
iot-firmware-research/docs/decisions/2026-03-25-choose-network-attack-surface.md
```

## 示例

### 示例 1：初始研究决策

```markdown
# Decision Record: 2026-03-25-001

## 基本信息
- **Decision ID**: 2026-03-25-001
- **时间**: 2026-03-25 14:30:00
- **决策者**: Research Lead AI

## 研究目标
发现 IoT 固件中的高危漏洞

## 背景
用户提供了某厂商的 IoT 设备固件，要求进行完整的安全审计。

## Agent Strategy
**Multi**

**选择理由**：
- 固件审计需要多个视角：逆向、攻击面、漏洞验证
- 认知视角冲突：状态机 vs 加密层
- 失败成本高：需要多个 Specialist Agent 并行探索

**Specialist Agents 分配**：
- **reverse-analyst**：固件解包、二进制分析、协议逆向
- **code-audit**：代码审计（网络接口、串口、存储）
- **skeptic**：反证初步假设、挑战结论

## 备选路径分析

### 路径 A：网络攻击面
- **描述**：分析网络服务，寻找远程漏洞
- **优势**：影响大，可远程利用
- **风险**：可能防护较好
- **可行性**：中

### 路径 B：本地攻击面
- **描述**：分析串口、本地存储
- **优势**：防护较弱，容易发现
- **风险**：需要物理访问
- **可行性**：高

### 路径 C：固件更新
- **描述**：分析固件更新机制
- **优势**：可能获得 RCE
- **风险**：可能加密签名
- **可行性**：中

## 最终路径
**选择**: 同时探索路径 A、B、C（并行）

**选择理由**：
- 三个攻击面相对独立，可并行
- 失败成本高，需要风险对冲
- Multi-Agent 可以同时探索

## 风险评估
- **技术风险**：固件可能加密，需要解密
- **时间风险**：完整审计可能耗时较长
- **缓解措施**：先快速识别加密，如加密则先解密

## 证据收集计划
- 固件文件系统结构
- 网络服务代码审计
- 串口协议分析
- 漏洞 PoC 验证

## 预期产出
- 完整的安全审计报告
- 高危漏洞 PoC
- 修复建议

## 下一步
1. 固件解包
2. 识别文件系统和二进制文件
3. 调用 Specialist Agents 开始分析
```

### 示例 2：路径变化决策

```markdown
# Decision Record: 2026-03-25-002

## 基本信息
- **Decision ID**: 2026-03-25-002
- **时间**: 2026-03-25 16:45:00
- **决策者**: Research Lead AI

## 研究目标
调整研究方向：从网络攻击面转向固件更新机制

## 背景
在网络服务审计中未发现明显漏洞。reverse-analyst 发现固件更新协议存在签名验证绕过的可能性。

## Agent Strategy
**Single**

**选择理由**：
- 目标已明确（固件更新协议）
- 单一攻击面
- 不需要多 Agent 协作

## 备选路径分析

### 路径 A：继续网络审计
- **描述**：深入分析网络服务代码
- **优势**：可能发现逻辑漏洞
- **风险**：已审计 2 轮，发现概率低
- **可行性**：低

### 路径 B：转向固件更新
- **描述**：分析固件更新协议
- **优势**：初步发现签名验证问题
- **风险**：需要逆向协议
- **可行性**：高

## 最终路径
**选择**: 路径 B（转向固件更新）

**选择理由**：
- 网络审计已无新发现
- 固件更新存在明显漏洞迹象
- ROI 更高

## 风险评估
- **技术风险**：协议可能复杂
- **缓解措施**：使用 reverse-analyst 逆向协议

## 证据收集计划
- 固件更新协议逆向
- 签名验证代码分析
- PoC 编写

## 预期产出
- 固件更新漏洞 PoC
- 漏洞复现视频

## 下一步
1. 抓取固件更新流量
2. 逆向协议格式
3. 分析签名验证逻辑
```

---

## 关键原则

1. **必须包含 Decision ID**：格式为 YYYY-MM-DD-XXX
2. **必须说明 Agent Strategy**：Single 或 Multi
3. **必须分析备选路径**：至少 2 个路径
4. **必须说明选择理由**：为什么选这个路径
5. **必须评估风险**：技术风险、时间风险
6. **必须明确下一步**：具体的行动

---

## 审查清单

提交 Decision Record 前检查：

- [ ] Decision ID 格式正确
- [ ] Agent Strategy 已明确
- [ ] 备选路径已分析（至少 2 个）
- [ ] 选择理由清晰
- [ ] 风险已评估
- [ ] 下一步明确
- [ ] 保存到正确位置（`xxx-research/docs/decisions/`）
